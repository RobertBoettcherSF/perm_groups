# Permutation Groups in Ada SPARK

## Overview

This repository contains an Ada SPARK implementation of Donald E. Knuth's algorithms Aₖ(π) and Bₖ(π) for computing strong generators and transversal systems of permutation groups. The implementation is designed to be formally verifiable using GNATPROVE and follows the algorithms described in Knuth's *The Art of Computer Programming*, Volume 2, Section 4.6.3.

## Features

- **Formal Verification**: All core algorithms are implemented with SPARK 2022 contracts (preconditions, postconditions, loop invariants) for formal verification with GNATPROVE.
- **Knuth's Algorithms**: Complete implementation of algorithms Aₖ(π) and Bₖ(π) for permutation group computation.
- **Data Structures**: Efficient implementation using Ada.Containers.Vectors for dynamic storage of permutations.
- **Comprehensive Testing**: Test suites for sparse examples, dense examples, and random permutations.
- **Examples**: Demonstrations of various permutation groups from Knuth's paper.

## Mathematical Background

### Permutation Groups

A permutation group is a set of permutations of a finite set that is closed under composition and taking inverses. The algorithms in this repository compute:

1. **Strong Generators**: A set of generators that can generate the entire group efficiently
2. **Transversal Systems**: Data structures that enable efficient membership testing

### Knuth's Algorithms

**Algorithm Aₖ(π)**: Appends a new permutation π to T(k) and ensures the data structure remains up-to-date so that Γ(k) equals the new group generated by T(k).

**Algorithm Bₖ(π)**: Ensures π is in Γ(k) and the data structure remains up-to-date of order k-1.

### Membership Test

The membership test determines if a permutation π ∈ Π(k) is in Γ(k):
- Let π map k ↦ j
- If σₖⱼ is empty, π ∉ Γ(k)
- If k = 1, π ∈ Γ(k)
- Otherwise, π ∈ Γ(k) if and only if πσₖⱼ⁻¹ ∈ Γ(k-1)

## Compilation Instructions

### Prerequisites

- **GNAT Ada Compiler**: Version supporting Ada 2022 and SPARK
- **GPRbuild**: GNAT Programming Studio build tool
- **GNATPROVE**: For formal verification (optional for compilation)

### Building the Project

1. **Clone the repository**:
   ```bash
   git clone https://github.com/RobertBoettcherSF/perm_groups.git
   cd perm_groups
   ```

2. **Build the library and executables**:

3. **Build specific executables**:
   ```bash

   gprbuild -P perm_groups.gpr test_random
   gprbuild -P perm_groups.gpr knuth_examples
   ```

4. **Run the examples**:
   ```bash
   ./bin/test_sparse
   ./bin/test_dense
   ./bin/test_random
   ./bin/knuth_examples
   ```

## Verification Instructions

### Running GNATPROVE

To formally verify the implementation using GNATPROVE:

1. **Verify all source files**:
   ```bash
   gnatprove -P perm_groups.gpr --mode=all
   ```

2. **Verify specific files**:
   ```bash
   gnatprove -P perm_groups.gpr src/permutations.adb
   gnatprove -P perm_groups.gpr src/sims_algorithm.adb
   ```

3. **Use the configuration file**:
   ```bash
   gnatprove -c gnatprove.config -P perm_groups.gpr
   ```

4. **Generate proof reports**:
   ```bash
   gnatprove -P perm_groups.gpr --report=all
   ```

### Verification Levels

The implementation supports different verification levels:

- **Level 0**: Type checking only
- **Level 1**: Flow analysis
- **Level 2**: Flow analysis + assertions
- **Level 3**: Full proof with pre/post conditions
- **Level 4**: Full proof with all contracts (recommended)

## Performance Notes

### Complexity Analysis

- **Algorithm Aₖ(π)**: O(k² |T(k)| |Σ(k)|) in the worst case
- **Algorithm Bₖ(π)**: O(k²) recursive calls in the worst case
- **Membership Test**: O(k²) for testing if π ∈ Γ(k)

### Memory Usage

The implementation uses:
- **T(k)**: Stores strong generators at each level k
- **Σ(k,j)**: Stores transversal elements for each k and j
- Memory complexity: O(k² |G|) where |G| is the group size

### Optimization Considerations

1. **Index Range**: The `Max_Size` constant in `permutations.ads` can be adjusted based on your needs. Larger values will use more memory.

2. **Recursion Depth**: The algorithms use recursion, so very large groups might hit stack limits. Consider increasing the stack size if needed.

3. **Caching**: GNATPROVE can cache proof results to speed up subsequent verifications.

### Performance Tips

- For large groups, consider using a smaller `Max_Size` during development
- Use the Release build mode for performance-critical applications
- The algorithms are designed to be efficient for groups up to several hundred elements
